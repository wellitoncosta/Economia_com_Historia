------------------------------------
- GET  /api/quiz/salas
  Lista todas as salas de quiz existentes na base de dados.

- GET  /api/quiz/salas/{id}/perguntas
  Lista perguntas publicas da sala sem expor respostaCorreta.

- GET  /api/quiz/salas/{id}/ranking
  Lista ranking da sala.

- POST /api/quiz/salas/{id}/entrar
  Agora aceita visitante sem JWT. Quando nao ha utilizador autenticado, o backend usa o utilizador tecnico visitante@local.

- POST /api/quiz/salas/{id}/responder
  Nova rota HTTP para responder quiz sem depender do WebSocket.
  Publica para visitante e autenticados.
  Request JSON:
  {
    "perguntaId": "uuid-da-pergunta",
    "resposta": "Texto da alternativa escolhida",
    "tempoGastoMs": 12000
  }
  Response JSON:
  {
    "perguntaId": "uuid-da-pergunta",
    "correta": true,
    "pontos": 10,
    "pontuacaoAcumulada": 30
  }
  Observacoes:
  - Se a sala estiver AGUARDANDO, o backend muda automaticamente para EM_ANDAMENTO ao responder.
  - Se o participante ainda nao existir, o backend cria automaticamente.
  - Sala FINALIZADA nao aceita resposta.

QUIZ - CRIACAO E CONTROLE
-------------------------
- POST /api/quiz/salas continua exigindo JWT, mas deixou de exigir MASTER/moderador.
- Qualquer utilizador autenticado pode criar sala de quiz.
- POST /api/quiz/salas/{id}/perguntas exige JWT.
- POST /api/quiz/salas/{id}/iniciar e /finalizar exigem JWT.

RECUPERACAO DE CONTA
--------------------
- POST /api/recuperacao/solicitar valida se o email existe na plataforma.
- Se o email nao existir, retorna erro 404 RESOURCE_NOT_FOUND.
- Se SMTP estiver configurado por SMTP_USERNAME/SMTP_PASSWORD, envia email real.
- Se SMTP nao estiver configurado ou falhar, o backend devolve mensagem com codigo de teste para desenvolvimento:
  {
    "mensagem": "Servico de email nao configurado. Codigo de teste: 123456"
  }
- POST /api/recuperacao/validar valida o codigo sem inutiliza-lo.
- POST /api/recuperacao/redefinir-senha consome o codigo e altera a palavra-passe.

FRONTEND E ORIGEM DOS DADOS
---------------------------
- Listagem de quiz usa GET /api/quiz/salas.
- Execucao de quiz usa GET /api/quiz/salas/{id}/perguntas e POST /api/quiz/salas/{id}/responder.
- Rankings usam GET /api/rankings/regioes e GET /api/rankings/instituicoes.
- Conteudos e recomendacoes nao usam imagens externas mockadas; os cards sao construidos com titulo, tipo, categoria, descricao e datas vindas da API/base de dados.

DOCUMENTACAO DOS MAPPINGS DO SISTEMA
====================================

Projeto: plataforma-aprendizagem
Base URL padrao: http://localhost:8080
Formato padrao: application/json

Como os JSON funcionam neste projeto
------------------------------------
1. Os requests com body usam DTOs Java do tipo record. Cada parametro do record vira uma chave JSON.
   Exemplo: record LoginRequest(String email, String password) vira:
   {
     "email": "user@email.com",
     "password": "12345678"
   }

2. Campos anotados com:
   - @NotBlank: obrigatorios, nao podem vir vazios.
   - @NotNull: obrigatorios, nao podem ser null.
   - @Email: precisam ter formato de email.
   - @Size: precisam respeitar tamanho minimo/maximo.
   - @Min: precisam ser maiores ou iguais ao valor definido.
   - @Pattern: precisam respeitar o padrao indicado.

3. Enums devem ser enviados como texto exatamente igual ao nome do enum:
   - Role: VISITANTE, INSCRITO, CRIADOR, REVISOR, MASTER
   - TipoConteudo: TEXTO, VIDEO, AUDIO
   - PapelForum: DONO, MODERADOR, MEMBRO
   - EstadoSalaQuiz: AGUARDANDO, EM_ANDAMENTO, FINALIZADA
   - TipoEntidadeVoto: TOPICO, COMENTARIO
   - TipoVoto: UP, DOWN

4. Datas retornam em formato ISO-8601, por exemplo:
   "2026-07-03T20:30:00"

5. Rotas protegidas precisam de JWT no header:
   Authorization: Bearer <token>

6. Estrutura padrao de erro:
   {
     "codigo": "VALIDATION_ERROR",
     "mensagem": "Pedido invalido",
     "timestamp": "2026-07-03T20:30:00",
     "path": "/api/auth/login",
     "detalhes": [
       "email: must not be blank"
     ]
   }

Codigos comuns de erro:
- 400 BUSINESS_RULE: regra de negocio violada.
- 400 VALIDATION_ERROR: JSON invalido ou campos obrigatorios ausentes.
- 403 FORBIDDEN: utilizador sem permissao.
- 404 RESOURCE_NOT_FOUND: recurso nao encontrado.
- 500 INTERNAL_ERROR: erro interno.

1. AUTH
=======

1.1 POST /api/auth/register
---------------------------
Descricao: registra um novo utilizador.
Autenticacao: publica.
Status de sucesso: 200 OK.

Request JSON:
{
  "email": "user@email.com",
  "password": "12345678",
  "regiao": "Luanda",
  "instituicao": "ISPTEC"
}

Regras do request:
- email: obrigatorio, formato de email.
- password: obrigatorio, minimo 8 e maximo 100 caracteres.
- regiao: opcional.
- instituicao: opcional.

Response JSON:
{
  "token": "jwt-token",
  "utilizador": {
    "id": "uuid-do-utilizador",
    "email": "user@email.com",
    "role": "VISITANTE",
    "pontosAcumulados": 0,
    "regiao": "Luanda",
    "instituicao": "ISPTEC"
  }
}

Como funciona:
- O backend cria o utilizador e devolve um token JWT.
- O objeto utilizador mostra os dados publicos do utilizador autenticado.

1.2 POST /api/auth/login
------------------------
Descricao: autentica um utilizador existente.
Autenticacao: publica.
Status de sucesso: 200 OK.

Request JSON:
{
  "email": "user@email.com",
  "password": "12345678"
}

Regras do request:
- email: obrigatorio, formato de email.
- password: obrigatorio.

Response JSON:
{
  "token": "jwt-token",
  "utilizador": {
    "id": "uuid-do-utilizador",
    "email": "user@email.com",
    "role": "INSCRITO",
    "pontosAcumulados": 25,
    "regiao": "Luanda",
    "instituicao": "ISPTEC"
  }
}

Como funciona:
- Se as credenciais estiverem corretas, retorna o token JWT.
- Use esse token no header Authorization nas rotas protegidas.

10. QUIZ - WEBSOCKET/STOMP
==========================

Endpoint WebSocket configurado:
- /ws

10.1 Cliente envia resposta
---------------------------
Destino STOMP:
/app/quiz/{salaId}/responder

Descricao: envia a resposta de uma pergunta do quiz.
Autenticacao: o userId e obtido da sessao WebSocket pelo backend.

Payload enviado pelo cliente:
{
  "perguntaId": "uuid-da-pergunta",
  "resposta": "Java",
  "tempoGastoMs": 12000
}

Regras do payload:
- perguntaId: obrigatorio.
- resposta: obrigatorio.
- tempoGastoMs: obrigatorio.

10.2 Resposta individual enviada pelo servidor
----------------------------------------------
Destino STOMP para o utilizador:
/user/queue/quiz/respostas

Payload recebido:
{
  "perguntaId": "uuid-da-pergunta",
  "correta": true,
  "pontos": 10,
  "pontuacaoAcumulada": 30
}

Como funciona:
- Depois de responder, o backend calcula se esta correta, quantos pontos ganhou e a pontuacao acumulada.

10.3 Ranking em tempo real enviado pelo servidor
------------------------------------------------
Destino STOMP publico da sala:
/topic/quiz/{salaId}/ranking

Payload recebido:
[
  {
    "utilizadorId": "uuid-do-utilizador",
    "pontuacao": 30
  }
]

Como funciona:
- Depois de cada resposta, o backend publica o ranking atualizado da sala.
